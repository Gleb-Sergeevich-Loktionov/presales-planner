# Спецификация — Планировщик пресейла (planner_dev)

Версия: 2026-06-08. Основано на ответах продукт-овнера + текущем коде planner_dev.
Статус: целевая модель для переработки. Что переиспользуем / что меняем — раздел 11.

---

## 0. Критический разбор (поправки после сверки с кодом)

- **C1 (ключевая).** «Бот формирует задачу из NL-описания» требует LLM-ключа (Claude).
  Fallback-парсер (`infra/llm/basic.py`) — regex, умеет только фикс-команды и проект
  ИЗ ШАБЛОНА; превратить свободное описание в задачу он НЕ может. Ключей пока нет.
  **Решение:** шаблонный поток planner (уже готов, 359 тестов) = рабочий MVP без ключа;
  **NL-интейк — отдельная фаза под ключ**, а не часть базового MVP.
- **C2.** Слой способностей (роли→скиллы) работает на шаге ПРЕДЛОЖЕНИЯ исполнителя,
  ДО солвера. Солвер получает задачи с уже подтверждённым исполнителем (`Assignment`).
- **C3.** Оценку часов задачи даёт LLM, человек валидирует — это судьба, не матем. солвера.
- **C4.** В planner уже есть strictness A/B/C на шаблон-задачах. Новый слой Role/Skill
  сосуществует: скиллы — общий подбор; strictness — быстрый ярлык внутри шаблона.
- **Готово в planner:** Person/Template/TemplateTask(+Assignee strictness, Dependency
  FS/SS)/Project/Task(status, is_preliminary)/Assignment/Dependency/DayOverride/
  PlanVersion(proposed/committed)/AuditLog. **Нет:** Role/Skill/RoleSkill/PersonRole,
  NL-интейк, TaskHistory.

## 1. Что строим

Telegram-бот — «виртуальный менеджер» агентства пресейла. Люди в проектном
чате описывают задачи **своими словами** (текст или голос), бот превращает их в
структурированные задачи, **подбирает исполнителя** по способностям и загрузке,
**считает расписание** под дедлайны/зависимости/capacity и **сигналит**, когда не
помещается. Вся математика — детерминированный код (солвер). LLM — только для
(а) разбора естественной речи и (б) человеко-понятных объяснений. В базу пишет
**только бот**; люди управляют через чат. Каждое изменение — в журнал.

## 2. Пользователи

- **Команда (свои):** Иван, Мария, Дарья, Софья, Ольга, Пётр. Читают/спрашивают/
  гоняют «что-если» — все. Создавать/менять план — только **админы**.
- **Аутсорс-дизайнеры** (Елена и др.): внешний пул, без capacity, фикс-срок на
  дизайн-задачи, нужны готовые тексты на вход (зависимость). Не назначаются как
  обычные исполнители.
- **Клиент (заказчик услуг)** — НЕ сущность системы. Сроки вносит команда.

## 3. Основной поток

```
1. Человек кидает в чат (текст/голос): «надо сделать X к пятнице»
2. Бот (LLM) разбирает → черновик задачи: название · оценка срока · нужные скиллы ·
   зависимости · предлагаемый исполнитель
3. Валидация: автор (или админ) правит/подтверждает черновик
4. Назначение: автор указал кого ИЛИ бот предложил (скиллы + загрузка) → подтверждение
5. Солвер ставит подтверждённые задачи в расписание (capacity, зависимости, дедлайн)
6. Разрушающая правка → бот переспрашивает → коммит в базу + запись в журнал
```

Дневная сводка по загрузке постится в чат утром. «Что-если» считается на копии,
базу не трогает, возвращает дифф.

## 4. Модель данных

**Справочник (способности):**
- **Person** — name, capacity_h (дефолт 8, настраивается), is_admin, is_active,
  is_external, telegram_id.
- **Role** — name, description.
- **Skill** — name, description (что это / зачем / как используется).
- **RoleSkill** — роль → её стандартные скиллы (скиллы определяются ролью:
  роль «программист» → стандартный набор скиллов программиста).
- **PersonRole** — человек → его роли. Способности человека = объединение скиллов
  его ролей. (Плюс при желании личные скиллы в обход роли — опционально.)

**Задачи и план:**
- **Project** — name, deadline (опц.), brief_return_date (опц.), template (опц.),
  status.
- **Task** — name, est_hours, required_skills[], status (not_done / done /
  preliminary / confirmed), is_splittable, allow_two_assignees, project_id,
  fixed_* (для done/immovable), source ('bot_formed' | 'template').
- **Dependency** — task → depends_on, link_type FS (жёсткая) / SS (мягкая, внахлёст).
- **Assignment** — задача → человек → даты + по-дневная раскладка (выход солвера).
- **DayOverride** — человек × день × capacity (0 = отпуск/выходной).
- **TaskHistory** — выполненные задачи человека (для будущего подбора по истории).
- **Audit** — кто/что/когда менял (журнал).
- **PlanVersion** — proposed / committed (черновик vs зафиксированный план).

**Шаблоны (быстрый старт, опц.):** именованные наборы (standard / lite) как
заготовка — можно подгрузить в проект и править. Не обязательны.

## 5. Подбор исполнителя

Боту нужно «может ли человек сделать задачу»:
1. **Сейчас:** задача → нужные скиллы (LLM по описанию) → люди, чьи роли покрывают
   эти скиллы → среди них по загрузке → предложение → подтверждение человеком.
2. **Позже:** добавляется сигнал по **истории задач** (что человек делал раньше).
   На старте история пуста, накапливается.

Способности — мягкая подсказка для предложения, не жёсткое ограничение. Автор
всегда может назначить вручную.

## 6. Время и перегруз

- **Гибрид:** бот оценивает в **часах** внутри (точная математика солвера), людям
  показывает **дни/окна** (напр. «дизайн ≈ 2 рабочих дня»). Не таймер над людьми.
- **Capacity:** дефолт 8 ч/день на человека, настраивается. Точные часы заполнит
  команда.
- **Перегруз — мягкий:** можно превысить, бот подсвечивает и предлагает рычаги
  (lite / +человек / сдвиг). Решают люди. Целевой кейс сигнала — 2–3 проекта на
  один дедлайн.
- Рабочий календарь РФ (праздники/выходные) — через isdayoff (уже есть).

## 7. Дедлайны

- **Прямой режим (основной):** есть дедлайн → строим план назад от срока.
- **Обратный режим:** дедлайна нет → бот считает самую раннюю реальную дату +
  буфер **+2 рабочих дня**, менеджер подтверждает.
- **Дедлайн первичен:** при нестыковке система не молчит — сигналит и предлагает
  рычаги, но не двигает молча.

## 8. Команды бота

Интерфейс — **кнопки + короткие команды**, естественный язык поверх. **Голос** —
в MVP (Whisper), но включается при наличии OpenAI-ключа; без ключа — текст.

- Создать задачу (из описания) · подтвердить черновик · назначить/переназначить
- Создать проект (опц. из шаблона) · подтвердить план
- Спросить загрузку («что у Ольга на неделе?») · перегрузы
- «Что-если» (на копии) · отметить отпуск · запустить пересчёт (replan)
- Дневная сводка (авто, утром)

## 9. Права и запись

- Чтение/вопросы/что-если — всем. Менять план — только админ.
- Разрушающие правки — через подтверждение бота.
- В базу пишет только бот (single-writer); конфликтов нет.
- Журнал всех изменений (actor + время).

## 10. Стек

Python 3.12 · aiogram 3 · FastAPI+htmx (админка) · SQLAlchemy 2 async · Postgres 16 ·
Redis 7 · NetworkX (жадный солвер + критический путь) · Claude (intent, с fallback на
простой парсер без ключа) · Whisper/OpenAI (голос, опц.) · APScheduler (сводка) ·
matplotlib (Gantt/heatmap) · uv. Гексагональная архитектура (bot/web/app/domain/infra),
адаптеры за портами.

## 11. Что переиспользуем / что перерабатываем

**Переиспользуем (готово в planner):** солвер (жадный, критический путь, diff/
what-if), бот-каркас и хендлеры, FastAPI-админка, БД-слой+alembic, intent-парсер +
fallback, Whisper STT, APScheduler, рендер Gantt/heatmap, 359 тестов.

**Перерабатываем:**
1. **Интейк задач:** сейчас только фикс-шаблон → добавить «бот формирует задачу из
   NL-описания + валидация автором». Шаблон демотируется в «быстрый старт».
2. **Слой способностей:** сейчас `role_label` строкой → ввести Roles/Skills/RoleSkill/
   PersonRole (модель из lattice). Бот подбирает исполнителя по скиллам.
3. **Назначение:** сейчас жёсткая strictness-матрица в шаблоне → автор/бот-предложение
   по способностям+загрузке + подтверждение. Strictness опционально остаётся как
   подсказка.
4. **TaskHistory** — новая таблица (задел под подбор по истории).

**lattice_dev — заморожен.** Его работа (роли→скиллы) портируется в planner как слой
способностей.

## 12. Границы

**В MVP:** интейк из NL + валидация, подбор по скиллам+загрузке, солвер, прямой+
обратный режим, перегруз-сигнал+рычаги, что-если, дневная сводка, отпуска, журнал,
права, голос (при ключе), быстрый-старт шаблоны.

**vNext (не сейчас):** OR-Tools/тяжёлый солвер, мульти-тенант, множитель сложности
проекта, подбор по истории задач (задел кладём, логику потом), Notion/CRM интеграция,
прод-деплой как отдельная фаза перед пилотом.

## 13. Открытое (заполняет команда, не блокирует архитектуру)

- Точные capacity по людям · telegram_id админов · API-ключи (Claude/OpenAI) ·
  стандартный набор скиллов на каждую роль · фикс-срок дизайн-задач · содержимое
  быстрый-старт шаблонов (standard/lite) · сервер для деплоя.

## 14. План работ (фазы)

База: шаблонный поток planner уже работает без ключа — это стартовый MVP.

- **Ф0 (без ключа, стартуем).** Слой способностей: модели Role/Skill/RoleSkill/
  PersonRole + миграция + seed (команда, роли, стандартные скиллы ролей, привязка людей).
- **Ф1 (без ключа).** Подбор исполнителя по скиллам+загрузке как ПОДСКАЗКА на шаге
  назначения в существующем потоке + поток подтверждения. (Капабилити-матчинг детерминирован.)
- **Ф2 (НУЖЕН Claude-ключ).** NL-интейк: свободное описание → LLM-черновик задачи
  (название/оценка часов/нужные скиллы/зависимости) → валидация автором. Новый
  `CreateTaskIntent` + хендлер. Без ключа фича выключена, шаблонный поток работает.
- **Ф3.** Голос (нужен OpenAI-ключ), дневная сводка, что-если, отпуска — проверить на
  новой модели (в planner уже есть, сверить).
- **Ф4.** TaskHistory (задел) + подбор по истории — логика позже.
- **Ф5.** Приёмочные сценарии в тест-чате → деплой на VPS → пилот.

Зависимости от ключей: Ф0–Ф1 работают без ключей; Ф2 (NL-интейк) и голос — под ключи.
